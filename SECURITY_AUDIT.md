# Auditoria de Segurança — GestCondo

Data: 2026-07-06. Âmbito: código-fonte completo em `app/`, `components/`, `lib/`. Sem acesso a infraestrutura de produção (hosting, DNS, WAF, backups) — essas camadas não são avaliadas aqui e têm de ser auditadas separadamente quando existirem.

Convenção de gravidade: **Crítica** (exploração direta, dados de outros ou perda de dados) · **Alta** (exploração plausível ou ausência de controlo essencial) · **Média** (fraqueza que agrava outro problema ou UX de segurança pobre) · **Baixa** (robustez/boas práticas).

## 1. Autenticação

| # | Achado | Gravidade | Detalhe | Recomendação |
|---|--------|-----------|---------|--------------|
| S1 | ~~Sem verificação de email~~ **Resolvido 2026-07-06** | Alta | `emailAndPassword.requireEmailVerification: true` em `lib/auth.ts` — sign-in é recusado até o email ser confirmado. `emailVerification.sendOnSignUp`/`sendOnSignIn: true` enviam o email automaticamente. UX tratada em `components/auth-form.tsx` (ecrã "Falta só um passo" após registo, em vez de redirecionar silenciosamente). **Nota:** sem `RESEND_API_KEY` configurada, o email não é enviado de facto — fica só registado na consola do servidor (ver S2/`lib/email.ts`), suficiente para testar localmente mas **obrigatório configurar uma chave real antes de qualquer utilizador real se registar**, ou ninguém consegue confirmar a conta. |
| S2 | ~~Sem recuperação de password~~ **Resolvido 2026-07-06** | Alta | Fluxo completo: `/esqueci-password` (pedir link) → email com link → `/redefinir-password?token=...` (definir nova password) → `authClient.resetPassword`. Envio via `lib/email.ts`, um `fetch` direto à API REST do Resend (sem SDK) — se `RESEND_API_KEY` não estiver definida, cai em modo local (consola do servidor), nunca falha a operação principal. Mesma nota que S1: **precisa de uma `RESEND_API_KEY` real antes de produção**, senão os utilizadores nunca recebem o email de facto. |
| S3 | Sem MFA | Média | Só email+password. Para contas de administração de condomínio (acesso a dados financeiros e pessoais de todos os condóminos) isto é insuficiente para um produto profissional. | Adicionar 2FA (TOTP) pelo menos para o perfil admin, via plugin do better-auth. |
| S4 | ~~Política de password fraca e só client-side~~ **Parcialmente resolvido 2026-07-06** | Média | `emailAndPassword.minPasswordLength: 10` configurado no servidor em `lib/auth.ts` (antes só existia `minLength={8}` no `<input>`, contornável). Falta ainda bloqueio de passwords triviais/reutilizadas (have-i-been-pwned check) — não implementado. | Considerar integração com a API do Have I Been Pwned para recusar passwords conhecidas como comprometidas. |
| S5 | ~~Rate limiting não configurado explicitamente~~ **Parcialmente resolvido 2026-07-09** | Média | `rateLimit` configurado explicitamente em `lib/auth.ts` (ativo também em dev, não só produção), com regras mais restritas em `/sign-in/email`, `/sign-up/email` e `/forget-password` (5 pedidos/minuto). `storage` continua no default (`memory`) — não sobrevive a restart nem é partilhado entre instâncias serverless. | Migrar `storage` para `'database'` (tabela própria do better-auth) ou Redis/Upstash antes de produção com múltiplas instâncias. |
| S6 | Segredo de sessão | Alta | Não existe `.env` no repositório (correto), mas também não há validação de arranque que force a existência de `BETTER_AUTH_SECRET` em produção — o better-auth já falha com erro claro se usar o secret por omissão (confirmado durante o build desta auditoria), o que é positivo, mas depende de configuração manual do operador. | Documentar (README/`.env.example`) as variáveis obrigatórias: `DATABASE_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`. Adicionar `.env.example`. |
| S7 | Cookie `sameSite: 'none'` forçado em dev | Baixa | `lib/auth.ts` força `sameSite: 'none', secure: true` quando `NODE_ENV === 'development'` (necessário para o iframe de preview do v0). Está corretamente isolado por `NODE_ENV`, mas é um ponto a confirmar sempre que o processo de deploy for definido, para garantir que nunca corre em produção. | Confirmar em CI/CD que `NODE_ENV=production` está sempre definido no build de produção. |

## 2. Autorização e modelo de acessos

| # | Achado | Gravidade | Detalhe | Recomendação |
|---|--------|-----------|---------|--------------|
| S8 | ~~Apenas 2 perfis existem; o pedido do produto exige 7~~ **Resolvido 2026-07-06** | Crítica (para o objetivo do produto) | `Perfil` (agora em `lib/perfis.ts`) passou a `'admin' \| 'gestor' \| 'condomino' \| 'inquilino' \| 'fornecedor' \| 'auditor'`, mais `user.superAdmin` (flag de plataforma, ortogonal ao `perfil` por condomínio) — os 7 papéis pedidos. Grupos de permissão (`PERFIS_GESTAO`, `PERFIS_CONSULTA_GESTAO`, `PERFIS_ACESSO_FINANCEIRO`) controlam o quê cada um vê/escreve: inquilino e fornecedor deixaram de ver dados financeiros/patrimoniais (`getMovimentos`, `getFracoes`, dashboard); auditor vê tudo mas não pode escrever em lado nenhum (`requireMembroComEscrita`); só admin/gestor podem administrar. **Continua a faltar** (fora do âmbito desta correção): fluxo de onboarding para uma empresa de administração gerir vários condomínios na prática (hoje só existe um condomínio por instalação — ver S9), e qualquer workflow real de atribuição a fornecedores (o perfil existe, o fluxo de atribuição de ocorrências/orçamentos não). | Ver `FUNCTIONAL_GAPS.md`. |
| S9 | ~~Sem isolamento multi-condomínio (tenant isolation)~~ **Resolvido e verificado 2026-07-06** | Crítica (para o objetivo do produto) | Adicionada tabela `condominio` e coluna `condominioId` (FK `NOT NULL`) em todas as tabelas de dados do condomínio. Todas as queries de leitura, escrita, atualização e eliminação em `app/actions/*.ts` e no dashboard filtram agora por `condominioId` — incluindo as operações que recebem um `id` do cliente (`eliminarMovimento`, `atualizarEstadoOcorrencia`, etc.), que antes só filtravam por `id` e seriam um IDOR entre condomínios assim que existisse mais do que um. **Verificado com um teste de integração real** (`lib/db/tenant-isolation.dbtest.ts`, corrido via `pnpm test:db` contra a base de dados PostgreSQL real do utilizador, dentro de uma transação sempre revertida) — confirma que (1) uma leitura filtrada por `condominioId` nunca devolve linhas de outro condomínio, e (2) uma eliminação por `id` que reproduz exatamente o `where` de `eliminarMovimento` falha em afetar um registo de outro condomínio, mesmo com o `id` certo. **Continua a faltar** (fora do âmbito desta correção): um fluxo de convite/criação de um segundo condomínio pela UI — hoje só existe um único condomínio por instalação, criado automaticamente no primeiro registo (`lib/session.ts:getMembroAtual`). O isolamento está pronto e verificado para quando esse fluxo existir; o fluxo em si ainda não existe. | Construir o fluxo de criação/convite de condomínio antes de operar mais do que um condomínio na mesma base de dados (ver `FUNCTIONAL_GAPS.md`). |
| S10 | ~~Corrida de duplicação de `membro`~~ **Resolvido 2026-07-06** | Média→Baixa | Índice único `(userId, condominioId)` em `membro` (evita linhas duplicadas). **Mais grave, e só descoberto ao testar contra uma base de dados real pela primeira vez nesta sessão:** o "bootstrap" do primeiro condomínio (`getMembroAtual()`) era um check-then-act sem qualquer serialização — dois pedidos concorrentes no primeiro acesso de sempre a um sistema novo podiam cada um deixar de ver o condomínio do outro e criar **dois condomínios "primeiros" distintos** (reproduzido com 8 pedidos em paralelo: sem a correção, 2 condomínios; com a correção, sempre exatamente 1). Corrigido envolvendo a secção crítica numa transação com `pg_advisory_xact_lock`, que serializa pedidos concorrentes. Confirmado com um teste de concorrência real (8 pedidos em paralelo → 1 condomínio, 1 membro). | Nenhuma — comportamento verificado. |
| S11 | `requireAdmin()` não valida `estado` | Baixa | Documentado como aceite: promover um `membro` pendente a `perfil: 'admin'` não o aprova automaticamente; o gate na layout continua a bloqueá-lo (falha fechada, não é um buraco de segurança), mas é uma armadilha de UX/operação para o admin que o promover. | Ao promover a admin, definir `estado: 'aprovado'` na mesma operação (`atualizarPerfilMembro`). |
| S12 | ~~FK "suave" sem `.references()`~~ **Resolvido 2026-07-06** | Baixa | `movimento.fracaoId` agora referencia `fracao.id` com `onDelete: 'set null'`. |
| S13 | ~~Exposição de contactos entre condóminos~~ **Resolvido 2026-07-07** | Média (também RGPD, ver `GDPR_CHECKLIST.md`) | `getFracoes()` só devolve `contactoEmail`/`contactoTelefone` reais a quem tem `temConsultaGestao()` (admin, gestor, auditor, super admin) — condómino/inquilino/fornecedor recebem esses campos como `null`. A coluna "Contacto" em `/fracoes` fica escondida por completo para quem não tem essa permissão, em vez de mostrar sempre "—". Verificado com um teste de integração real (`lib/db/contactos-privacidade.dbtest.ts`). | Nenhuma — considerar no futuro tornar a partilha do próprio contacto opt-in por condómino, se houver necessidade de os condóminos se contactarem entre si. |

## 3. Vulnerabilidades clássicas (OWASP)

| Classe | Estado | Nota |
|---|---|---|
| SQL Injection | ✅ Não encontrada | Todo o acesso a dados usa Drizzle ORM com parâmetros (`eq`, `and`, `.values()`). Não há SQL em texto livre em lado nenhum do código auditado. |
| XSS | ✅ Não encontrada | React escapa JSX por omissão; não existe `dangerouslySetInnerHTML` em todo o código. O único vetor plausível (`documento.url` renderizado como `href`) já valida `^https?://` em `criarDocumento` (app/actions/documentos.ts), fechando o vetor `javascript:`. |
| CSRF | ✅ Mitigado pela framework | Server Actions do Next.js validam a origem do pedido por omissão. `trustedOrigins` em `lib/auth.ts` está corretamente restrito a domínios Vercel/V0 — **tem de ser atualizado com o domínio de produção real quando este existir**, ou os logins em produção falham (ou, se configurado de forma demasiado permissiva "para resolver rápido", reabre risco de CSRF). |
| IDOR | ⚠️ Parcialmente coberto, cobertura de testes ainda parcial | As ações que existem verificam propriedade corretamente (ex.: `eliminarOcorrencia` em `app/actions/ocorrencias.ts` restringe condóminos ao seu próprio `userId`, e todas as escritas por id filtram também por `condominioId`). **Atualizado 2026-07-06:** introduzido `vitest`, mas os testes escritos até agora cobrem a matriz de permissões pura (`lib/perfis.test.ts`) e formatação (`lib/format.test.ts`) — ainda **não existem testes de integração contra uma base de dados real** a exercitar as próprias server actions/IDOR entre condomínios, porque este ambiente de desenvolvimento não tem Postgres disponível (ver `MVP_PLAN.md`). Prioridade seguinte assim que houver uma BD de teste. |
| Uploads inseguros | N/A — ainda não implementado | Não existe nenhuma funcionalidade de upload de ficheiros hoje (`documento.url` é apenas um link de texto). É um requisito funcional planeado (atas, faturas, fotos de ocorrências) com risco de segurança próprio — ver recomendações abaixo antes de implementar. |
| Permissões excessivas | ⚠️ Ver S13 | Contactos pessoais partilhados além do necessário. |
| Exposição de ficheiros/segredos | ✅ Não encontrada | Sem `.env` no repositório; `.gitignore` cobre `.env*.local`. **Gap**: não cobre um simples `.env` sem sufixo — adicionar por precaução, mesmo sem repositório git ainda inicializado. |

## 5. Logging e auditabilidade

| # | Achado | Gravidade | Recomendação |
|---|--------|-----------|--------------|
| S17 | ~~Nenhum registo de auditoria aplicacional~~ **Resolvido 2026-07-06** | Alta (também requisito legal, ver secção 4 do pedido original) | Nova tabela `audit_log` (`lib/db/schema.ts`) escrita por `lib/audit.ts:registarAuditoria()` em todas as ações sensíveis: criar/eliminar movimento, alternar pago, criar/eliminar aviso/documento/fração, aprovar/rejeitar/atualizar/mudar perfil de membro, criar/atualizar estado/eliminar ocorrência. | S18 | Sem logging estruturado nem correlação de pedidos | Baixa | Não há nenhuma camada de logging (nem `console.error` nas server actions em caso de erro — os erros só chegam ao `toast` no browser). Dificulta diagnóstico de incidentes e deteção de abuso. | Adicionar logging estruturado no servidor (sem incluir dados pessoais em texto livre) com um id de correlação por pedido. |

## 6. Resumo de recomendações prioritárias de segurança

1. ~~**Bloqueador de produto (crítico):** implementar isolamento multi-condomínio (S9)~~ — **Resolvido 2026-07-06** ao nível do modelo de dados e das queries. Falta ainda o fluxo de produto para efetivamente criar/gerir um segundo condomínio (ver S9 e `FUNCTIONAL_GAPS.md`).
2. ~~**Bloqueador de confiança (alto):** recuperação de password + verificação de email (S1, S2)~~ — **Resolvido 2026-07-06.** Falta configurar uma `RESEND_API_KEY` real antes de produção (hoje os emails só ficam na consola do servidor).
3. ~~**Auditoria/accountability (alto):** `audit_log` + soft-delete em dados financeiros~~ — **Resolvido 2026-07-06** (S17). Atas/deliberações ainda não existem (módulo de Assembleias por construir).
4. **Modelo de papéis (crítico para o objetivo do produto):** redesenhar para os 7 perfis pedidos (S8).
5. Cabeçalhos de segurança, remoção de `ignoreBuildErrors`, lint funcional (S14–S16).
6. ~~MFA para admin, rate limiting explícito, política de password no servidor (S3–S5)~~ — rate limiting **resolvido parcialmente 2026-07-09** (S5); MFA (S3) e verificação de password comprometida (resto de S4) continuam por fazer.
7. ~~Restringir exposição de contactos pessoais (S13)~~ — **Resolvido 2026-07-07.**

Nenhuma das vulnerabilidades OWASP clássicas testadas (SQLi, XSS, CSRF) foi encontrada no código atual — o que existe está bem construído dentro do seu âmbito reduzido. O risco de segurança desta aplicação está concentrado em **ausência de controlos** (auditoria, recuperação de conta, isolamento multi-tenant, MFA) mais do que em código vulnerável.
